## Retrospective

### What went well

- The team kept the main telemetry pipeline working.
- Docker Compose made the project easier to run and test.
- The dashboard became more useful and readable.
- We adapted the scope after feedback and removed unstable features.
- Customer feedback helped us focus on realistic priorities.

### What did not go well

- Some features were started before the final product scope was clear.
- The Gate component took time but was later removed from the final delivery.
- Performance and bottleneck issues were discovered late.
- Testing with realistic traffic should have started sooner.

### What we learned

- A stable MVP is better than many unfinished features.
- Deployment requirements should be clarified early.
- Real traffic tests are important, not only simple or synthetic tests.
- Handover documentation should be maintained during development.

### Action points for next sprint

- Add database support for historical traffic data.
- Improve documentation and startup instructions.
- Investigate the traffic processing bottleneck.
- Test with more realistic traffic sources.
